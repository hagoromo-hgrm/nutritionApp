# AGENTS.md

## プロジェクト概要

このリポジトリは、iPhoneのホーム画面から利用する個人用栄養管理PWAを開発するためのものです。要件の正本は [`docs/requirements.md`](docs/requirements.md) とします。

アプリは、食事記録を短時間で入力できること、オフラインで継続利用できること、ユーザーが自分でデータをバックアップ・復元できることを優先します。

## 開発環境と標準構成

- OS: macOS
- エディタ: Visual Studio Code
- フロントエンド: Vite + TypeScript + React
- ローカルデータベース: IndexedDB（Dexie.jsを使用）
- PWA: Web App Manifest + Service Worker（`vite-plugin-pwa`）
- バーコード: `@zxing/browser`等のブラウザー対応ライブラリ
- 食品データ変換: Python
- 本番配信: Cloudflare Pagesを第一候補、GitHub Pagesを代替候補とする

実際の依存関係やスクリプトが未作成の場合は、既存の要件を保った最小構成から追加します。不要なフレームワークやバックエンドを導入しないでください。

## 開発オーケストレーション

- オーケストレータは、ユーザーの要望とリポジトリの現状を確認し、具体的な実装案の提示、作業分解、サブエージェントへの指示、成果の統合と検証を担う。
- 単純な実装・修正には `Luna-high` または `Luna-xhigh` を使用し、分析や高度な推論を要する作業には `Sol-high` または `Sol-xhigh` を使用する。
- サブエージェントの速度は標準を使用する。通常は `Sol` の利用を必要最小限に抑え、作業の難度に見合うモデルを選ぶ。
- commitは作業上必要であれば行ってよい。pushはユーザーが明示した場合に限って行う。
- サブエージェントへ、対象ファイル、変更してよい範囲、変更してはいけない範囲、受け入れ条件、検証方法を具体的に伝える。統合前にオーケストレータが差分を確認する。

## 実装上の必須方針

### データ保存

- 食品マスター、食事記録、設定、お気に入りはIndexedDBに保存する。
- CookieやlocalStorageに食事履歴の正本を保存しない。
- IndexedDBのスキーマにはバージョンを付け、将来のマイグレーションを可能にする。
- 保存処理はトランザクションで行い、失敗時に成功扱いにしない。
- 食事記録には、登録時点の食品名・基準量・栄養値のスナップショットを保存する。食品マスターの後日編集で過去の集計が意図せず変わってはいけない。
- 食品を削除しても食事記録は削除しない。削除済み食品として履歴を表示できるようにする。

### オフラインとPWA

- 初回アクセス後にアプリ本体をキャッシュし、オフラインでも保存済み食品の検索、記録、編集、削除、集計を利用可能にする。
- 外部商品APIが使えない場合も、バーコード番号を保持した手入力登録へ切り替えられるようにする。
- HTTPS、Manifest、Service Worker、iPhoneホーム画面起動を実機で確認する。
- アプリ更新時に古いキャッシュが残る問題を考慮し、更新通知または更新確認を実装する。

### 栄養計算

- 基本式は `摂取栄養値 = 基準量あたりの栄養値 × 摂取量 ÷ 基準量` とする。
- g、ml、個、袋などの単位を混同しない。単位変換できない値は推測で変換しない。
- 欠損値を自動的にゼロと解釈しない。未集計・データ不足を区別する。
- 正規の栄養集計は、寄与する記録に欠損値があれば `null` を維持する。グラフでは正規集計を書き換えず、既知の寄与分だけを合計して棒を描く。全寄与が欠損する項目は棒を描かず、欠損数値は `--.-` と表示する。
- 保存値と表示値を分け、画面表示は小数1桁を初期値とする。CSVでは必要な精度を保持する。
- 医療的な診断、治療、個別の栄養指導を行わない。

### 食品データ

- 一般食品は日本食品標準成分表を元データとし、Pythonでアプリ用形式へ変換する。
- 変換処理には、元データの版、取得日、出典、加工日を記録する。
- ユーザー提供または事前検証済みの食品DBは、元ファイルを直接利用せず、再生成可能な検証・正規化処理を通した生成物だけを `imported` として投入する。基準量や全栄養値が欠ける名称候補をゼロ補完して投入しない。
- 市販品はパッケージの栄養成分表示またはユーザー入力を正とする。
- Open Food Facts等の外部データは補助情報として扱い、取得後にユーザーが確認してから保存する。
- 外部データの正確性・完全性を保証しないことを画面上で示す。
- ライセンス、帰属表示、データベース共有条件を確認せずに外部データを同梱・再配布しない。

### バーコード

- JAN/GTINは文字列として扱い、先頭ゼロを失わない。
- カメラ権限を拒否された場合でも、番号手入力で同じ処理を実行できるようにする。
- 端末内検索、外部API検索、手入力登録を一連のフォールバック導線にする。
- 外部APIへ送る情報は原則としてバーコード番号のみとする。

### バックアップと出力

- JSONバックアップには食品、食事記録、設定、お気に入り、データ形式バージョンを含める。
- JSON復元前に現在データを自動退避し、不正ファイルでは既存データを変更しない。
- 復元方式の初期値は全置換とし、上書き警告を表示する。
- CSVは閲覧・分析および食事履歴の移送用であり、食品・設定等を含む完全バックアップの復元形式にはしない。
- CSVの初期列順は要件定義書の `FR-013` に従う。UTF-8 BOM付きCSVを基本とする。
- 個人の食事データやバックアップをGitへコミットしない。

## MVPの優先順位

最初に次の順で実装します。

1. IndexedDBの型・スキーマ・マイグレーション
2. 一般食品の初期投入と食品検索
3. 市販品の手入力登録
4. 食事登録、編集、削除、当日集計
5. 最近使った食品、お気に入り、前日コピー
6. PWAキャッシュとオフライン動作
7. JSONバックアップ・復元、CSV出力
8. バーコード読み取り、端末内検索、外部API連携

OCR、ログイン、端末間同期、共有機能はMVP完成後の拡張です。MVPの操作性やデータ保全を損なう変更を先に追加しないでください。

## UI・操作方針

- 日本語UIを使用する。
- iPhone縦画面と片手操作を優先する。
- 既存食品は、検索・最近使った食品・お気に入りから少ない操作で登録できるようにする。
- 食事区分は朝食、昼食、夕食、間食を初期値とする。
- FOODMASTERの管理画面にはQUICK ADDを表示しない。FOODSは「あ・か・さ・た・な・は・ま・や・ら・わ・英数字／その他」等の五十音見出しでまとめ、各見出しを展開できるようにする。食事登録時のQuick導線は維持する。
- 料理メニュー編集は、基本情報、追加済み構成食材、追加検索を平坦で判別しやすい領域に分け、入れ子の枠を重ねない。
- 食品詳細・登録画面には料理メニューとの紐付けタブを設けない。料理メニューとの関係は料理メニュー側で構成食材として管理する。
- メニューセット編集は登録済みメニュー全件のチェック一覧にせず、「追加済み項目」と「食品・料理メニューを追加する領域」に分ける。
- 食事登録のメニュータブはメニューセットを最上位にし、その後に主食、主菜、副菜等の料理カテゴリを並べる。
- 今日の食事カードには食品名・分量・kcalだけを表示し、区分単位の「編集」から確認画面へ移動する。食品単位の編集・削除と、料理メニュー記録の構成食材変更は確認画面で行う。設定画面の説明アイコンは各操作行の右端列を揃える。
- 食事の表示順は摂取日時と分離して保存し、並び替え操作は食事区分の確認画面だけに置く。タッチドラッグと上下ボタンの両方を利用可能にし、ドラッグ中は指に追従する浮動プレビューを表示する。ドラッグ開始は専用ハンドルに限定し、開始後の移動・終了は画面全体で受け取って必ず状態を解放する。
- グラフの数値列は開始位置を揃え、既知分があれば小計を表示する。全寄与が欠損する場合だけ `--.-` とする。適正範囲内の数値は青、それ以外は単位と同じ文字色にする。
- グラフの1週間・1ヶ月・3ヶ月・1年タブは1画面に表示する日数を決める。過去方向への横スクロールは制限せず、必要な履歴を遅延読込して位置を維持する。棒は下から朝食・昼食・夕食・間食の順に積み上げ、棒間隔は暫定的に1日枠の5分の1とする。
- TODAY概要と詳細で食事区分色・適正範囲色・欠損仕様を統一する。間食色は `#C9FFA6`、TODAY概要のカロリー目標線は左から75%とする。間食詳細には基準量を表示しない。
- 保存、削除、復元など状態を変える操作は、完了または失敗を明確に通知する。
- 復元、全削除、前日コピーなど誤操作の影響が大きい操作には確認を設ける。
- エラー文は原因と次に取れる操作を日本語で示す。

## コード品質

- TypeScriptのstrict設定を基本とし、`any`を安易に使用しない。
- UI、IndexedDB、栄養計算、バックアップ、外部APIを分離する。
- 日時はISO 8601形式で保存し、食事日の判定は `Asia/Tokyo` を基準とする。
- ID生成、単位処理、栄養計算、CSVエスケープ、JSON検証は再利用可能な関数にする。
- 開発完了の明示後に既存データ形式を変更する場合は、マイグレーションとバックアップ互換性を確認する。開発段階の扱いは「変更時の注意」に従う。
- 大量データをメモリへ無制限に読み込まず、IndexedDBの検索・インデックスを活用する。
- コメントは「何をしているか」ではなく、仕様上の理由や注意点を補足するために使用する。

## 検証手順

変更後は、利用可能な範囲で次を実行します。

```bash
npm run lint
npm run typecheck
npm test
npm run build
```

スクリプトが存在しない場合は、無理に追加せず、実際に定義されているコマンドを確認して実行します。Pythonの構文確認では、macOSのキャッシュ書き込み問題を避けるため、必要に応じて書き込み可能なキャッシュ先を指定します。

最低限、次の手動確認を行います。

- iPhone Safariからホーム画面へ追加して起動できる。
- 初回読み込み後に通信を切っても、既存食品の検索・記録・集計ができる。
- バーコードのカメラ拒否時に手入力へ進める。
- 不正JSONの復元で既存データが壊れない。
- JSON出力後に復元すると、食品・食事・設定・お気に入りが戻る。
- CSVの日本語、列順、数値、改行を確認できる。

## 変更時の注意

- `docs/requirements.md` は要件の正本であり、実装都合で黙って仕様を縮小・変更しない。差異が必要な場合は、変更理由を記録する。
- 重要な設計判断と廃止した方針は `docs/decisions.md` に記録する。
- ユーザーが開発完了を明示するまでは開発段階とし、過去のログ、旧バックアップ、旧データ形式との互換性は実装判断を阻害する制約にしない。必要なスキーマ変更、再投入、開発用データの破棄を選択できる。
- 開発段階であっても、作業範囲外の現行ユーザーデータや個人データを削除、移行、上書きしない。対象に含まれる現行データへ破壊的変更が必要な場合は、対象を明確にし、必要な保護手段を確認する。
- 外部API、ライセンス、ブラウザー仕様など変更され得る事項は、実装時点の公式情報を確認する。
- 作業範囲外のファイルや既存のユーザー変更を上書きしない。
- 新しい依存関係は、目的、ライセンス、バンドルサイズ、オフライン動作への影響を確認してから追加する。

## 完成判定

MVPは、iPhoneのホーム画面から起動でき、オフラインで既存食品の食事記録と当日集計ができ、JSONバックアップ・復元とCSV出力ができる状態を完成とします。食品情報の完全性や自動推定の高度さより、毎日継続できる入力速度とデータ保全を優先します。
