# 個人用栄養管理PWA 要件定義書

- 文書版数: 0.1
- 作成日: 2026-07-15
- 対象: 個人利用
- 想定端末: iPhone（Safariからホーム画面に追加）
- 想定開発環境: macOS、Visual Studio Code、Python、JavaScript/TypeScript

## 1. 目的

日常の食事を、スマートフォンから短時間で記録し、1日単位の栄養摂取量を確認できる個人用アプリを作成する。

巨大な食品データベースを最初から構築するのではなく、一般食品の成分データと、自分がよく購入・摂取する市販品を蓄積していく方式とする。App Storeへの公開や常時稼働するバックエンドを必須にせず、低コストで運用できることを重視する。

## 2. 基本方針

1. iPhoneのホーム画面から起動できるPWAとして提供する。
2. 食事記録と食品データは端末内のIndexedDBに保存する。
3. アプリ本体は静的ホスティングで配信する。
4. オフラインでも、既存食品の検索・食事記録・集計を利用できるようにする。
5. 食品情報が見つからない場合は、ユーザーが手入力して登録できるようにする。
6. データ消失に備え、JSONバックアップと復元を初期版から提供する。
7. 記録操作の短さを、機能数より優先する。

## 3. 想定ユーザーと利用場面

### 3.1 想定ユーザー

- 開発者本人が主利用者である。
- 日本国内で販売される食品を主に利用する。
- 1台のiPhoneを主端末とする。
- 将来的に別端末でも利用する可能性はあるが、初期版では自動同期を必須としない。

### 3.2 主な利用場面

- 食事直後に、食品と量を登録する。
- よく食べる食品を履歴またはお気に入りから再登録する。
- 市販品のバーコードを読み取り、商品情報を検索または新規登録する。
- その日のエネルギー、たんぱく質、脂質、炭水化物などを確認する。
- 必要に応じてデータをJSONまたはCSVで書き出す。

## 4. スコープ

### 4.1 MVPに含める機能

- PWAとしての起動・インストール
- オフライン利用
- IndexedDBへの端末内保存
- 一般食品データの初期登録
- 市販品の手入力登録・編集
- 食品名検索
- 食品の分量入力
- 朝食、昼食、夕食、間食への食事登録
- 当日の食事一覧表示、編集、削除
- エネルギー、たんぱく質、脂質、炭水化物、食物繊維、食塩相当量の集計
- 最近使った食品の表示
- お気に入り登録
- 前日の食事のコピー
- バーコードのカメラ読み取り
- バーコードに紐づく食品の検索・登録
- JSONバックアップ、JSON復元
- 食事履歴のCSV出力
- 栄養目標値の設定と当日値との比較
- エラー表示、入力値の最低限の検証

### 4.2 MVPに含めない機能

- ユーザーアカウント、ログイン
- 複数端末間の自動同期
- SNS共有や他ユーザーとのデータ共有
- 医療目的の診断・治療・食事指導
- 写真だけによる料理名・分量・栄養値の自動推定
- 大規模な商品マスターの継続的な自動収集
- 有料サービスや広告の導入

### 4.3 将来拡張

- Supabase、Firebase、Cloudflare D1などを使った端末間同期
- レシピと材料の一括登録
- 料理単位のお気に入り登録
- 週次・月次の推移グラフ
- 栄養素ごとの目標値・上限値の詳細管理
- OCRによる栄養成分表示の入力補助
- 食品データのインポート・更新画面
- Appleヘルスケア等との連携検討
- ログイン、複数ユーザー、共有機能

## 6. 機能要件

### FR-001 PWA起動

- HTTPSで配信する。
- Web App Manifestを提供する。
- iPhoneのSafariからホーム画面に追加できる。
- ホーム画面から起動した場合、ブラウザーのアドレスバーを表示しないアプリ表示を目標とする。
- アプリアイコンと表示名を設定する。

### FR-002 オフライン動作

- 初回アクセス後、アプリ本体のHTML、CSS、JavaScript、アイコンをService Workerでキャッシュする。
- オフライン時も、端末内に保存済みの食品の検索、食事記録、編集、削除、集計を利用できる。
- オフライン時に外部商品APIへ接続できない場合は、理由を表示し、手入力登録へ誘導する。
- 新しいアプリ版がある場合は、次回起動時などに更新を通知する。

### FR-003 食品検索

- 食品名、メーカー名、バーコードで検索できる。
- 前方一致または部分一致を基本とする。
- 検索結果には食品名、メーカー、基準量、主な栄養値を表示する。
- 最近使った食品とお気に入りを通常検索より短い操作で選択できる。
- 初期版では表記揺れの高度な自動吸収は必須としない。

### FR-004 一般食品データ

- 日本食品標準成分表を元データとする。
- Pythonの変換スクリプトで、アプリに必要な項目をJSONまたはCSVへ変換する。
- 食品コード、食品名、食品群、基準量、エネルギー、たんぱく質、脂質、炭水化物、食物繊維、食塩相当量を保持する。
- 欠損値はゼロと断定せず、未設定として扱う。集計時の扱いはデータ項目ごとに定義する。
- データの出典名、版、取得日または加工日を保持する。

### FR-005 市販品の登録

- 商品名、メーカー、バーコード、内容量、基準量、栄養値を登録できる。
- 栄養成分表示が「1個当たり」「1袋当たり」「100g当たり」などの場合、基準量と単位を明示して保存する。
- 登録後は食品検索、最近使った食品、お気に入りから利用できる。
- 同じバーコードが既に登録されている場合は、新規重複作成ではなく既存データの確認・編集を促す。
- ユーザー登録データには、入力元を `user` として記録する。

### FR-006 バーコード連携

- カメラ権限を要求し、一次元バーコードを読み取る。
- 読み取ったJAN/GTINを食品マスターから検索する。
- 端末内にない場合は、設定した外部商品APIへ問い合わせる。
- 外部APIから商品が見つかった場合は、取得情報を表示し、ユーザーの確認後に端末へ保存する。
- 外部APIから見つからない場合、または通信に失敗した場合は、バーコードを保持したまま手入力登録へ進める。
- カメラを使えない環境でも、バーコード番号の手入力で検索できる。
- API提供元のデータの正確性を保証しないことを画面上で明示する。

### FR-007 食事記録

- 食品を選択し、分量を入力して食事を登録できる。
- 食事区分は朝食、昼食、夕食、間食から選択する。
- 摂取日時を保存する。初期値は現在日時とする。
- 量は正の数値とし、0以下や異常に大きい値は警告する。
- 登録時点の栄養値を記録に保持する。後から食品マスターを編集しても、過去の記録が意図せず変わらないようにする。
- 登録済みの食事を編集・削除できる。
- 日付を選択して過去の食事を閲覧できる。

### FR-008 前日の食事コピー

- 当日の入力画面から、前日の食事を一覧表示できる。
- 前日の食事を全件または選択した食事区分単位でコピーできる。
- コピー時に日時を当日の現在時刻へ置き換える。
- コピー後は通常の食事記録として編集・削除できる。
- 同じ操作を誤って複数回実行した場合に備え、確認表示を行う。

### FR-009 栄養集計

- 選択日の合計を表示する。
- 初期表示項目は、エネルギー、たんぱく質、脂質、炭水化物、食物繊維、食塩相当量とする。
- 朝食、昼食、夕食、間食ごとの小計を表示する。
- 設定された目標値がある項目は、摂取量、目標量、達成率を表示する。
- 未設定値を含む栄養素は、ゼロとして断定せず、未集計またはデータ不足として扱う。
- 栄養値は原則として次の比例計算で求める。

  `摂取栄養値 = 基準量あたりの栄養値 × 摂取量 ÷ 基準量`

- 分量単位が個や袋の場合は、食品データに保存した1単位の重量または栄養値を使う。

### FR-010 最近使った食品・お気に入り

- 最近使った食品を、最後に食事登録した日時の降順で表示する。
- 最近使った食品の表示件数は初期値20件とする。
- 食品をお気に入りへ追加・解除できる。
- お気に入りは食品名の昇順を基本とする。
- 食品を削除しても、既存の食事記録は保持する。削除後の食品は「削除済み食品」として記録上のスナップショットを表示する。

### FR-011 目標値設定

- エネルギー、たんぱく質、脂質、炭水化物、食物繊維、食塩相当量の目標値を設定できる。
- 初期状態では、ユーザーが入力していない目標値は未設定とする。
- 医学的な推奨値を自動提示しない。目標値はユーザー自身が設定する。
- 目標値の単位を画面上に表示する。

### FR-012 JSONバックアップ・復元

- 食品マスター、食事記録、設定、お気に入り、料理メニュー、メニューセット、アプリのデータ形式バージョンをJSONで出力する。
- 出力ファイル名は `nutrition-backup-YYYYMMDD-HHmmss.json` を基本とする。
- JSON復元前に、現在のデータを上書きすることを警告する。
- 復元方式は初期値として「全置換」とする。
- JSONの形式、バージョン、必須項目を検証し、不正なファイルは取り込まない。
- 復元前に現在データの自動退避を作成する。
- 復元完了後、件数と最終バックアップ日時を表示する。

### FR-013 CSV出力

- 食事記録をCSVで出力できる。
- 初期列は次の順序とする。

  `id, date, eaten_at, meal_type, food_id, food_name, maker, barcode, amount, amount_unit, base_amount, base_unit, energy_kcal, protein_g, fat_g, carbohydrate_g, fiber_g, salt_g`

- UTF-8 BOM付きCSVを基本とし、日本語を表計算ソフトで開きやすくする。
- 出力対象日は、選択日または期間指定とする。期間指定の初期値は当月とする。
- 食品スナップショットの基準栄養値も出力し、このPWAで食事履歴を復元できる形式とする。
- CSV取り込み前に件数と同一IDの上書きを確認し、不正なCSVは既存データを変更せずに中止する。

## 7. データ要件

### 7.1 食品データ `foods`

| フィールド | 型 | 必須 | 内容 |
|---|---|---:|---|
| `id` | 文字列 | ○ | アプリ内で一意なID |
| `name` | 文字列 | ○ | 食品名・商品名 |
| `maker` | 文字列 | - | メーカー名 |
| `barcode` | 文字列 | - | JAN/GTIN。先頭ゼロを保持 |
| `source` | 列挙値 | ○ | `mext`, `open_food_facts`, `user` など |
| `source_version` | 文字列 | - | 元データの版やAPI由来情報 |
| `base_amount` | 数値 | ○ | 栄養値の基準量 |
| `base_unit` | 文字列 | ○ | `g`, `ml`, `個`, `袋` など |
| `serving_amount` | 数値 | - | 既定の摂取量 |
| `serving_unit` | 文字列 | - | 既定量の単位 |
| `nutrients` | オブジェクト | ○ | 栄養素と値 |
| `created_at` | ISO日時 | ○ | 作成日時 |
| `updated_at` | ISO日時 | ○ | 更新日時 |

### 7.2 食事記録 `meal_entries`

| フィールド | 型 | 必須 | 内容 |
|---|---|---:|---|
| `id` | 文字列 | ○ | 食事記録ID |
| `eaten_at` | ISO日時 | ○ | 摂取日時 |
| `meal_type` | 列挙値 | ○ | 朝食、昼食、夕食、間食 |
| `food_id` | 文字列 | ○ | 登録時に参照した食品ID |
| `food_snapshot` | オブジェクト | ○ | 登録時点の食品名・基準量・栄養値 |
| `amount` | 数値 | ○ | 摂取量 |
| `amount_unit` | 文字列 | ○ | 摂取量の単位 |
| `calculated_nutrients` | オブジェクト | ○ | 登録時点で計算した栄養値 |
| `created_at` | ISO日時 | ○ | 作成日時 |
| `updated_at` | ISO日時 | ○ | 更新日時 |

### 7.3 設定 `settings`

- 栄養目標値
- 表示単位
- 最終バックアップ日時
- データ形式バージョン
- 外部API利用設定（利用する場合）

### 7.4 IndexedDBの初期オブジェクトストア

- `foods`: 食品マスター
- `meal_entries`: 食事記録
- `favorites`: お気に入り食品ID
- `settings`: アプリ設定
- `metadata`: 初期データ投入済みフラグ、スキーマバージョンなど

食品名、バーコード、食事日時、食事区分には検索用インデックスを作成する。

## 8. 画面要件

### 8.1 今日の記録画面

- 今日の日付と合計栄養値を最上部に表示する。
- 朝食、昼食、夕食、間食ごとに登録内容と小計を表示する。
- 「食品を追加」「バーコードで追加」「前日をコピー」ボタンを配置する。
- 最近使った食品とお気に入りへ短い導線を設ける。

### 8.2 食品選択画面

- 検索欄、最近使った食品、お気に入り、検索結果を表示する。
- 食品を選択すると分量入力へ進む。
- 食品がない場合は「新しい食品を登録」へ進める。

### 8.3 分量入力画面

- 食品名、基準量、既定量、単位を表示する。
- 分量と食事区分を入力する。
- 登録前に今回の栄養値をプレビューする。

### 8.4 バーコード画面

- カメラ映像と読み取り状態を表示する。
- カメラ権限拒否時は、番号手入力へ切り替える。
- 端末内検索、外部API検索、手入力登録の結果を同じ導線で扱う。

### 8.5 食品管理画面

- ユーザー登録食品の追加・編集・削除を行う。
- 出典、基準量、単位、栄養値を確認できる。
- 食品削除時は食事履歴が残ることを説明する。

### 8.6 設定・データ管理画面

- 目標栄養値の設定
- JSONバックアップ出力
- JSON復元
- CSV出力
- データ件数、データ形式バージョン、最終バックアップ日時の表示
- アプリのバージョン表示

## 9. 非機能要件

### 9.1 操作性

- 今日の食事登録を、既存食品なら基本3操作以内で完了できることを目標とする。
- 片手操作を想定し、主要ボタンを画面下部または親指が届きやすい位置に置く。
- 入力欄はiPhoneのソフトウェアキーボードで入力しやすい型を指定する。
- 保存完了を画面上で明確に通知する。

### 9.2 性能

- 端末内の通常検索は、一般的な利用量で体感上すぐに結果が表示されること。
- 当日集計は、数百件程度の食事記録で実用的な待ち時間とする。
- 初期データ投入中は進捗を表示し、二重投入を防ぐ。

### 9.3 信頼性・復旧性

- 主要な書き込みはトランザクションで実行する。
- 破損や復元失敗時に、既存データを中途半端な状態にしない。
- JSON復元前に自動退避を作成する。
- 外部APIが停止していても、端末内機能は利用可能とする。

### 9.4 セキュリティ・プライバシー

- 食事記録を初期版では外部サーバへ送信しない。
- 静的ホスティングではHTTPSを使用する。
- 外部APIへ送信する情報は、原則としてバーコード番号のみとする。
- 個人情報、位置情報、連絡先へのアクセスを要求しない。
- 医療上の判断を行うアプリではない旨を表示する。

### 9.5 保守性

- TypeScriptを基本言語とし、主要なデータ型を定義する。
- 食品データ変換はPythonスクリプトとして分離する。
- IndexedDBのスキーマバージョンを管理し、将来の移行処理を用意できる構造にする。
- 外部APIの処理をアプリ本体から分離し、API変更の影響範囲を限定する。
- 個人データを含むバックアップファイルをGitへコミットしない。

## 10. 技術構成

### 10.1 推奨構成

- 開発OS: macOS
- エディタ: Visual Studio Code
- フロントエンド: Vite + TypeScript + React
- 端末内DB: IndexedDB + Dexie.js
- PWA: Web App Manifest + Service Worker + `vite-plugin-pwa`
- バーコード: `@zxing/browser`等のブラウザー対応ライブラリ
- データ変換: Python
- 静的ホスティング: Cloudflare Pages
- 代替ホスティング: GitHub Pages

### 10.2 開発時の構成

```text
project/
├─ src/
```

## 11. 外部データ・ライセンス方針

- 一般食品データは、利用条件を確認したうえで日本食品標準成分表を加工する。
- 元データの名称、版、URL、取得日、加工スクリプトの版を記録する。
- Open Food Factsを使う場合は、API利用条件、ODbL、帰属表示、データベース共有条件を確認する。
- 外部データは誤りや欠損があり得るため、取得値をユーザー確認なしに絶対的な正解として扱わない。
- 市販品の栄養値は、可能な限りパッケージの栄養成分表示を正とする。

## 12. エラー・例外処理

| 状況 | 処理 |
|---|---|
| IndexedDBへの保存失敗 | 保存失敗を通知し、再試行を促す。成功扱いにしない |
| カメラ権限拒否 | 番号手入力を表示 |
| バーコード未検出 | 再スキャンまたは手入力を案内 |
| 外部API通信失敗 | オフラインまたはAPIエラーを表示し、手入力へ誘導 |
| 食品が見つからない | バーコードを引き継いだ新規登録へ進む |
| JSON形式不正 | 取り込みを中止し、現在データを変更しない |
| JSON復元中断 | 復元前の自動退避から復旧できるようにする |
| 必須項目未入力 | 項目ごとのエラーを表示し、保存を禁止 |
| 不正な分量 | 数値・単位を確認し、保存を禁止 |
| 食品マスター削除済み | 食事記録のスナップショットを表示 |

## 13. 受け入れ条件

### MVP受け入れ条件

- iPhoneのSafariからサイトを開き、ホーム画面に追加できる。
- 一度読み込んだ後、通信を切っても既存食品の選択と食事登録ができる。
- 一般食品を検索し、分量を指定して朝食などへ登録できる。
- 市販品を手入力で登録し、次回検索から再利用できる。
- バーコードを読み取り、端末内検索または商品登録へ進める。
- 外部APIが利用できない場合でも、手入力登録に切り替えられる。
- 当日の合計と食事区分ごとの小計が表示される。
- 最近使った食品とお気に入りから食品を選べる。
- 前日の食事を当日へコピーできる。
- JSONを出力し、データを削除した状態から復元できる。
- CSVを出力し、指定した列順で食事記録を確認できる。
- JSON復元に失敗した不正ファイルで、既存データが壊れない。
- 主要な画面がiPhoneの縦画面で操作できる。

## 15. テスト方針

- 栄養計算: 基準量、分量、単位、丸め、欠損値をテストする。
- IndexedDB: 追加、編集、削除、検索、スキーマ更新をテストする。
- バックアップ: 正常JSON、不正JSON、古いバージョン、空データをテストする。
- CSV: 日本語、カンマ、改行、UTF-8 BOM、列順、食事履歴の復元をテストする。
- オフライン: 初回キャッシュ後に通信を切り、主要機能を操作する。
- バーコード: 読み取り成功、カメラ拒否、未知コード、通信失敗をテストする。
- iPhone実機: ホーム画面起動、縦画面、キーボード、カメラ、Safariストレージを確認する。
- データ消失対策: JSON復元前の自動退避と復旧手順を確認する。

## 16. 運用・バックアップ

- 初期版では、少なくとも定期的にJSONを手動保存する。
- バックアップファイルはiCloud Drive等、端末故障時にも取得できる場所へ保管する。
- バックアップを作成した日付をアプリ内に表示する。
- アプリ更新前や大規模な食品データ更新前にはバックアップを推奨する。
- 食品データの更新では、既存のユーザー登録食品と食事記録を上書きしない。

## 18. MVP完成の定義

次の状態をMVP完成とする。

> iPhoneのホーム画面から起動でき、通信がなくても、一般食品または一度登録した市販品を数回の操作で食事として記録し、当日の栄養合計を確認できる。データは端末内に保存され、JSONでバックアップ・復元でき、履歴をCSVで出力できる。

この段階では、食品情報の完全性や自動推定の高度さよりも、毎日の入力が継続できる操作性と、データを自分で管理できることを完成基準とする。
