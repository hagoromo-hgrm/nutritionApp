食品検索機能のマスターデータ整備について、以下の方針で設計・実装してください。

## 目的

日本食品標準成分表由来の食品データについて、一般ユーザーが検索・選択しやすい食品マスターを生成します。

主な生成対象は以下です。

- 一般ユーザー向けの表示名
- 読み仮名
- 検索用の別名
- 検索用の関連語
- 食品グループ
- 食品バリエーション属性

この処理には、LLMによる意味的な判断を積極的に使用してください。

特に、以下はルールだけで無理に処理せず、LLMが食品名の意味を判断して候補を生成する方針とします。

- 正式名称から自然な表示名への変換
- 読み仮名の生成
- 一般的な別名や表記揺れの生成
- 別名と関連語の区別
- 同一食品グループに属するかの判断
- 調理状態や皮の有無など、バリエーション属性の抽出

ただし、LLMの出力をそのまま本番の確定マスターへ反映しないでください。

基本方針は以下です。

```text
LLMによる候補生成
    ↓
構造化データの形式検証
    ↓
決定的なルールによる検証
    ↓
信頼度と警告による採用判定
    ↓
高信頼度は自動採用
中信頼度はレビュー対象
低信頼度は非採用または未統合
    ↓
確定済み食品マスター
```

## LLMの役割

LLMには、食品成分表の正式名称や同一カテゴリ内の周辺レコードを入力し、検索マスターデータの候補を生成させてください。

LLMが担当する処理は以下です。

1. 正式名称の意味解析
2. 一般ユーザー向け表示名の生成
3. 読み仮名の生成
4. 別名の生成
5. 関連語の生成
6. 食品グループ候補の生成
7. 食品バリエーション属性の抽出
8. 判断の信頼度と警告の出力

LLMには自由記述ではなく、JSONまたはJSONLによる構造化データを出力させてください。

## 重要な判断原則

### 食品グループ

食品グループは、ユーザーが通常、同じ基本食品として認識する単位とします。

以下のような差分は、原則として同一食品グループ内のバリエーションとして扱います。

- 生
- ゆで
- 焼き
- 蒸し
- 電子レンジ調理
- 皮つき
- 皮なし
- 脂身つき
- 脂身なし
- 水煮
- 油漬け
- 冷凍
- 乾燥

例：

```text
食品グループ: 鶏むね肉

バリエーション:
- 皮つき・生
- 皮なし・生
- 皮つき・焼き
- 皮なし・焼き
```

ただし、以下は原則として別の食品グループにしてください。

- 食品カテゴリが異なるもの
- 動物種、魚種、植物種が異なるもの
- 部位が異なるもの
- 基本食品が異なるもの
- 原材料と加工食品
- 加工によって独立した食品名称になっているもの
- 原材料構成が異なるもの
- 品種の違いが食品選択上重要なもの
- 栄養強化、栄養低減、商品特性が主要な違いとなるもの

別グループとする例：

```text
鶏むね肉
鶏もも肉
鶏ささみ
鶏レバー
```

```text
大豆
豆腐
納豆
豆乳
きな粉
```

```text
普通牛乳
低脂肪乳
無脂肪乳
```

```text
食塩
減塩食塩
```

```text
白ごはん
玄米ごはん
もち米ごはん
```

判断に迷う場合は、統合しないでください。

誤った分割は後から統合できますが、誤統合後に利用履歴や検索ログが蓄積すると分離が難しくなるため、食品グループについては保守的に判断してください。

### 表示名

正式名称は変更せず保持し、別フィールドとして一般ユーザー向け表示名を生成してください。

例：

```text
正式名称:
にわとり 若鶏肉 むね 皮なし 生

表示名:
鶏むね肉（皮なし・生）
```

```text
正式名称:
こめ 水稲めし 精白米 うるち米

表示名:
白ごはん
```

```text
正式名称:
調味料及び香辛料類 食塩類 食塩

表示名:
食塩
```

元の食品IDと正式名称は、必ず保持してください。

### 読み仮名

表示名および必要に応じて正式名称の読み仮名を生成してください。

読み仮名は検索用として利用します。

原則としてひらがなで生成し、記号や不要な空白を含めないでください。

特殊な魚介類、地域食品、品種名、専門用語など、読みが不確実な場合は警告を付け、レビュー対象にしてください。

### 別名と関連語

別名と関連語は明確に区別してください。

別名は、同一の食品を指す通称、略称、漢字表記、かな表記などです。

例：

```json
{
  "display_name": "鶏むね肉",
  "aliases": [
    "鶏胸肉",
    "とりむね肉",
    "とりむね",
    "トリムネ",
    "むね肉"
  ]
}
```

関連語は、同一食品ではないものの、検索語との関連がある語です。

例：

```json
{
  "display_name": "食塩",
  "aliases": [
    "塩",
    "しお"
  ],
  "related_terms": [
    "岩塩",
    "海塩",
    "減塩食塩"
  ]
}
```

以下のような食品を別名として扱わないでください。

- 食塩と減塩食塩
- 牛乳と低脂肪乳
- 白ごはんと玄米ごはん
- 大豆と納豆
- 豚肉とハム

## 出力形式

各食品について、少なくとも以下を含む構造化データを生成してください。

```json
{
  "source_id": "11234",
  "official_name": "にわとり 若鶏肉 むね 皮なし 生",
  "display_name": "鶏むね肉（皮なし・生）",
  "reading": "とりむねにくかわなしなま",
  "group": {
    "key": "meat:chicken:breast",
    "display_name": "鶏むね肉"
  },
  "variant": {
    "species": "鶏",
    "part": "むね",
    "skin": "皮なし",
    "preparation": "生",
    "processing": null,
    "variety": null
  },
  "aliases": [
    "鶏胸肉",
    "とりむね肉",
    "とりむね"
  ],
  "related_terms": [
    "鶏肉"
  ],
  "confidence": {
    "display_name": 0.98,
    "reading": 0.99,
    "grouping": 0.96,
    "aliases": 0.91,
    "related_terms": 0.85
  },
  "decision_codes": [
    "SAME_BASE_FOOD",
    "PART_RETAINED_IN_GROUP",
    "PREPARATION_MOVED_TO_VARIANT",
    "SKIN_STATE_MOVED_TO_VARIANT"
  ],
  "warnings": [],
  "needs_review": false,
  "generation_version": "1"
}
```

LLMによる長い自由記述の判断理由は不要です。

代わりに、短い判定理由コードを出力してください。

例：

```text
SAME_BASE_FOOD
SAME_SPECIES
PART_RETAINED_IN_GROUP
PREPARATION_MOVED_TO_VARIANT
SKIN_STATE_MOVED_TO_VARIANT
INDEPENDENT_PROCESSED_FOOD
DIFFERENT_MAIN_INGREDIENT
NUTRITIONAL_VARIANT_REQUIRES_SEPARATE_GROUP
UNKNOWN_NAME_TOKEN
AMBIGUOUS_READING
GROUPING_UNCERTAIN
```

## 信頼度

項目ごとに信頼度を出力してください。

ただし、LLM自身が出力した信頼度だけで採用可否を決めないでください。

採用判定には以下を組み合わせてください。

- LLMの信頼度
- JSON SchemaまたはPydanticの検証結果
- 決定的な禁止統合ルール
- 未解析語の有無
- 既知正解データとの整合性
- 同一グループ内の属性矛盾
- 前回生成結果との差分

初期基準は以下を目安としてください。

```text
読み仮名:
信頼度0.95以上かつ検証通過 → 自動採用

表示名:
信頼度0.95以上かつ警告なし → 自動採用

別名:
信頼度0.90以上かつ禁止条件なし → 自動採用候補

関連語:
検索順位への影響を弱くした上で採用可能

食品グループ:
信頼度0.98以上かつ全検証通過 → 自動採用
信頼度0.85以上0.98未満 → レビュー対象
信頼度0.85未満 → 自動統合しない
```

閾値は設定ファイルなどで変更可能にしてください。

## 機械検証

LLMの生成結果に対して、必ず検証スクリプトを実装してください。

### 形式検証

以下を検証してください。

- 元食品IDが存在する
- 元食品IDが重複していない
- 正式名称が保持されている
- 必須フィールドが存在する
- 信頼度が0以上1以下である
- 読み仮名が空でない
- 読み仮名が原則ひらがなである
- 食品グループに1件以上の食品が所属している
- JSON SchemaまたはPydanticモデルに適合する

### グループ検証

以下を検証してください。

- 異なる食品カテゴリを統合していない
- 異なる動物種、魚種、植物種を統合していない
- 明確に異なる部位を統合していない
- 原材料と加工食品を統合していない
- 栄養特性が異なる食品を統合していない
- 同一グループ内で属性が矛盾していない
- 未解析の名称要素が残る場合に自動確定していない

### 読み仮名検証

以下を検証してください。

- 不要な数字、記号、空白が残っていない
- 読み仮名が空でない
- 同一表示名に矛盾した読みが割り当てられていない
- 既知の主要食品の読みと矛盾していない
- 不確実な固有名詞や品種名に警告が付いている

### 別名検証

以下を検証してください。

- 別名と表示名が重複していない
- 別名同士が正規化後に重複していない
- 別食品を指す語が別名に入っていない
- 関連語を別名として登録していない
- 広すぎる一般語だけを別名として登録していない

## 禁止統合ルール

明確に同一グループへ統合してはいけない組み合わせを、コードへ直接埋め込まず、YAMLまたはJSONで管理してください。

例：

```yaml
forbidden_group_pairs:
  - [食塩, 減塩食塩]
  - [普通牛乳, 低脂肪乳]
  - [普通牛乳, 無脂肪乳]
  - [白ごはん, 玄米ごはん]
  - [精白米, 玄米]
  - [大豆, 納豆]
  - [大豆, 豆腐]
  - [豚肉, ハム]
  - [豚肉, ベーコン]
```

カテゴリ単位の禁止ルールや、部位違いの禁止ルールも設定可能にしてください。

## 既知正解データ

主要食品100件から200件程度について、手動で確認した既知正解データを用意してください。

既知正解データは以下に利用します。

1. LLMへ入力する具体例
2. 生成結果の回帰テスト
3. プロンプト変更時の品質比較
4. モデル変更時の品質比較
5. 自動採用閾値の調整

例：

```yaml
- source_id: "salt-001"
  official_name: "調味料及び香辛料類 食塩類 食塩"
  display_name: "食塩"
  reading: "しょくえん"
  aliases:
    - "塩"
    - "しお"
  related_terms:
    - "岩塩"
    - "海塩"
    - "減塩食塩"
  group_key: "seasoning:salt"

- source_id: "chicken-breast-raw-001"
  official_name: "にわとり 若鶏肉 むね 皮なし 生"
  display_name: "鶏むね肉（皮なし・生）"
  reading: "とりむねにくかわなしなま"
  group_key: "meat:chicken:breast"
```

## 実装要件

以下を実装してください。

1. 食品成分表の元データを読み込む処理
2. LLMへ渡す入力データの生成処理
3. LLMから構造化データを取得する処理
4. JSON SchemaまたはPydanticによる検証
5. 禁止統合ルールによる検証
6. 信頼度と警告による採用判定
7. レビュー対象のみを抽出する処理
8. 確定済みマスターを生成する処理
9. 既知正解データによる回帰テスト
10. 生成前後の件数と警告を出力するレポート
11. 前回の生成結果との差分レポート
12. 生成ルールおよびプロンプトのバージョン管理

本番アプリの検索リクエストごとにLLMを呼び出さないでください。

LLMは、食品マスターデータを生成・更新するオフライン処理または管理用バッチ処理で使用してください。

本番検索では、生成済みの表示名、読み仮名、別名、関連語、食品グループを使用し、決定的かつ高速に検索してください。

## 再現性と差分管理

以下の情報を生成結果に保存してください。

- 使用モデル
- プロンプトバージョン
- 生成ルールバージョン
- 実行日時
- 入力データのバージョン
- 生成元の食品成分表バージョン
- 採用判定の結果
- レビュー状態

同じ入力に対する出力が変化した場合に確認できるよう、前回結果との差分を出力してください。

差分には少なくとも以下を含めてください。

- 表示名の変更
- 読み仮名の変更
- 別名の追加と削除
- 関連語の追加と削除
- 食品グループの変更
- バリエーション属性の変更
- 信頼度の変化
- レビュー状態の変化

既に人間が承認または手動修正した値を、再生成時に無条件で上書きしないでください。

各項目について、生成元を管理してください。

例：

```text
llm
rule
manual
imported
```

原則として、`manual`の値は自動処理で上書きしないでください。

## レビュー対象

全件を人間が確認する設計にはしないでください。

以下に該当するものだけをレビュー対象として抽出してください。

- 食品グループの信頼度が閾値未満
- 読み仮名が不確実
- 未解析語が残っている
- 禁止統合ルールに近い候補
- 同一グループ内で属性が矛盾している
- 既知正解データと異なる
- 前回実行から食品グループが変更された
- 人間が修正した値とLLM出力が競合している
- 別名と関連語の区別が不明確
- 複数のグループ候補が同程度に妥当

レビュー対象は、CSV、JSONL、Markdownレポートなど、人間が確認しやすい形式で出力してください。

## テスト

少なくとも以下のテストを実装してください。

- 元食品IDと正式名称が保持される
- 「塩」が「食塩」の別名になる
- 「減塩食塩」が「食塩」の別名にならない
- 鶏むね肉の皮や調理状態違いが同一グループ候補になる
- 鶏むね肉と鶏もも肉が別グループになる
- 普通牛乳と低脂肪乳が別グループになる
- 白ごはんと玄米ごはんが別グループになる
- 大豆と納豆が別グループになる
- 読み仮名がひらがなで生成される
- 別名と関連語が重複しない
- 低信頼度の食品グループが自動採用されない
- 人間が修正した値が再生成で上書きされない
- 同じ入力データに対して安定した形式の出力が得られる

## 完了条件

以下を満たした時点で完了とします。

- LLMによる候補生成処理が実行できる
- 出力が構造化スキーマに適合する
- 禁止統合ルールが機能する
- 信頼度に応じて自動採用とレビュー対象を分離できる
- 主要な誤統合パターンをテストで検出できる
- 人間が修正したデータを再生成時に保護できる
- 既知正解データによる回帰テストが通る
- 生成結果と前回結果の差分を確認できる
- 食品検索要件書が更新されている
- 実装方法と実行方法がドキュメント化されている

実装後は、テストを実行し、生成件数、自動採用件数、レビュー対象件数、エラー件数、主要な警告内容を報告してください。
